# Phase 2 Implementation Plan — Accept Followers

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Enable Mastodon accounts to follow `@randomforms@happitec.com` by implementing HTTP Signature verification in the inbox handler and signed Accept-Follow delivery.

**Architecture:** Two new Lambda handlers — InboxHandler (receives Follow, verifies HTTP Signatures, stores follower, enqueues Accept) and DeliverHandler (SQS consumer, signs and POSTs Accept-Follow to remote inboxes). Shared HTTP Signature logic added to ActivityPubCore.

**Tech Stack:** Swift 6.3, aws-sdk-swift (AWSDynamoDB, AWSSSM, AWSSQS), swift-crypto (_CryptoExtras for RSA), AWS SAM

**Spec:** `docs/PROJECT-PLAN.md` (inbox handler lines 415-451, deliver handler lines 463-467, HTTP Signatures lines 419-429, DynamoDB schema lines 304-327, SAM IAM lines 1328-1343)

## Scope — Phase 2 Only

**Included:**
- InboxHandler: HTTP Signature verification (Cavage legacy), Follow activity handling, activity idempotency
- DeliverHandler: SQS consumer, sign outbound HTTP (Cavage legacy), POST Accept-Follow to remote inbox
- ActivityPubCore additions: HTTP Signature sign/verify, remote actor key fetching + DynamoDB caching, SQS delivery client
- SAM template: InboxFunction + DeliverFunction resources, IAM policies, SQS event source
- Follower storage in DynamoDB (FOLLOWER# entity, atomic followerCount increment)

**Deferred to later phases:**
- All inbox activity types except Follow + Undo-Follow (Phase 5)
- RFC 9421 signatures (start with Cavage only — wider compat)
- Shared inbox coalescing (Phase 3 — only matters for outbound post delivery, not Accept)
- Date header staleness check (nice-to-have, not blocking for Phase 2)

### Task 1: HTTP Signature verification (ActivityPubCore)

**Files:**
- Create: `Sources/ActivityPubCore/Crypto/HTTPSignature.swift`

- [ ] **Step 1: Implement Cavage HTTP Signature verification**

The `HTTPSignature` module needs:

1. `verify(headers:method:path:body:publicKeyPem:) throws -> Bool`
   - Parse the `Signature` header to extract `keyId`, `headers` (signed header list), `signature` (base64)
   - Reconstruct the signing string from the signed headers list:
     - `(request-target)` → `{method} {path}` (lowercase method)
     - Other headers → `{lowercased-name}: {value}`
     - Join with `\n`
   - Verify the `Digest` header matches SHA-256 of the body: `sha-256={base64(sha256(body))}`
   - Verify the RSA-SHA256 signature against the signing string using the public key
   - Use `_CryptoExtras` for RSA signature verification: `_RSA.Signing.PublicKey(pemRepresentation:)` then `.isValidSignature(_:for:padding:)`

2. `extractKeyId(from signatureHeader: String) -> String?`
   - Parse `keyId="..."` from the Signature header
   - This is used to look up the remote actor's public key

Key implementation detail: the `Signature` header format is:
```
Signature: keyId="https://remote.server/users/alice#main-key",headers="(request-target) host date digest",signature="base64..."
```

Parse by splitting on `,` then extracting key=value pairs (values are quoted).

- [ ] **Step 2: Implement Cavage HTTP Signature signing (for outbound)**

`sign(privateKeyPem:keyId:method:path:headers:body:) -> [String: String]`
- Constructs the signing string from the headers list
- Computes `Digest: sha-256={base64(sha256(body))}` for POST requests
- Signs with RSA-SHA256 using `_RSA.Signing.PrivateKey(pemRepresentation:)`
- Returns a dictionary of headers to add to the outbound request: `Signature`, `Date`, `Digest`, `Host`

```swift
    public let inbox: String
    public let sharedInbox: String?

    // ttl is handled as a DynamoDB attribute, not in the model
}
```

With `fromDynamoDB` and `toDynamoDB` conversion methods.

- [ ] **Step 2: Implement KeyManager**

`KeyManager` fetches and caches remote actor documents:

1. `getPublicKey(keyId:store:) async throws -> String`
   - Extract actor URI from keyId (strip `#main-key` suffix)
   - Check DynamoDB cache: `REMOTE_ACTOR#{actorUri}`, SK: `PROFILE`
   - If cached and not expired (TTL > now), return `publicKeyPem`
   - If not cached or expired, fetch the actor document via HTTPS (GET the actor URI with `Accept: application/activity+json`)
   - Parse the JSON to extract `publicKey.publicKeyPem`, `inbox`, `endpoints.sharedInbox`
   - Write to DynamoDB with 24h TTL
   - Return the public key PEM

2. `refreshKey(keyId:store:) async throws -> String`
   - Same as above but always fetches fresh (for key rotation retry)
   - Updates both `publicKeyPem` AND `ttl` in DynamoDB

Uses `Foundation.URLSession` for the HTTPS fetch (with `#if canImport(FoundationNetworking)` for Linux).

### Task 5: InboxHandler

**Files:**
- Create: `Sources/InboxHandler/main.swift`

- [ ] **Step 1: Implement InboxHandler**

The inbox handler receives POST requests from remote servers. For Phase 2, it handles `Follow` and `Undo` (of Follow). All other activity types → log + return 202.

Flow:
1. Extract `{username}` from path parameters
2. Verify actor exists in DynamoDB → 404 if not
3. Parse the request body as JSON
4. Extract the `Signature` header
5. Extract `keyId` from the signature → fetch remote actor's public key via `KeyManager`
6. Verify the HTTP Signature → 401 if invalid
7. On verification failure, try refreshing the key (key rotation) and verify again
8. Check activity idempotency — if activity `id` already stored, return 202 (dedup)
9. Route by activity `type`:
   - `Follow`:
     - Store follower record (actor URI, inbox URL, shared inbox from remote actor cache)
     - Increment follower count atomically
     - Build `Accept` activity JSON: `{"@context":"...","type":"Accept","actor":"our-actor-uri","object":{...original Follow activity...}}`
     - Enqueue delivery job to SQS (target: follower's inbox URL)
     - Return 202
   - `Undo`:
     - If object is a Follow (check `object.type` or resolve URI), remove follower, decrement count
     - Return 202
   - All other types → log and return 202

Environment variables: `TABLE_NAME`, `SERVER_DOMAIN`, `HANDLE_DOMAIN`, `QUEUE_URL`

IAM needs: DynamoDB (GetItem, PutItem, DeleteItem, UpdateItem, Query), SQS (SendMessage)

### Task 6: DeliverHandler (SQS consumer)

**Files:**
- Create: `Sources/DeliverHandler/main.swift`

- [ ] **Step 1: Implement DeliverHandler**

SQS consumer Lambda. Receives `SQSEvent` (from `AWSLambdaEvents`), processes each record:

1. Parse the SQS message body as `DeliveryJob`
2. Read the actor's private key from SSM Parameter Store (`/activity/{stage}/keys/{username}`)
3. Construct the outbound HTTP request:
   - Method: POST
   - URL: `job.targetInbox`
   - Body: `job.activityJSON`
   - Content-Type: `application/activity+json`
4. Sign the request with HTTP Signatures (Cavage format) using `HTTPSignature.sign(...)`
   - keyId: `https://{SERVER_DOMAIN}/users/{username}#main-key`
   - Sign headers: `(request-target) host date digest content-type`
5. Send the signed POST using `URLSession` (with `FoundationNetworking` on Linux)
6. Log the response status
7. If the response is 4xx (not retryable), log and move on
8. If 5xx or network error, throw to let SQS retry (visibility timeout handles this)

Environment variables: `TABLE_NAME`, `SERVER_DOMAIN`, `SSM_KEY_PREFIX` (from environment stack)

IAM needs: DynamoDB (GetItem, Query), SSM (GetParameter with decryption), SQS (implicit from SAM event source)

The handler uses `SQSEvent` from `AWSLambdaEvents`:
```swift
let runtime = LambdaRuntime {
    (event: SQSEvent, context: LambdaContext) in
    for record in event.records {
        // process each delivery job
    }
}
```
