# Nested Stacks, Selective Builds, and sam sync — Migration Design

## 2026-04-02 (revised 2026-04-03 after code review)

---

## Goal

Restructure the flat `activity-app-{stage}` SAM stack into nested CloudFormation stacks, implement selective Swift builds based on git diff, enable `sam sync --code` for rapid stage deploys, migrate live stacks without downtime, and lay groundwork for ephemeral PR environments.

---

## 1. Nested Stack Architecture

### 1.1 Design Rationale

The flat template contains 32 resources that have fundamentally different change frequencies:

- **Lambda functions** change on every code push (16 functions + their IAM roles auto-generated by SAM)
- **API Gateway** definitions change only when routes are added/removed
- **CloudFront distribution, cache policies, origin request policies, OAC, DNS** change only when caching strategy or domain config changes
- **S3 bucket policy** changes only if CloudFront distribution ID changes

Grouping by change frequency means a Lambda code change triggers an update to only the functions nested stack. CloudFormation skips unchanged nested stacks entirely.

### 1.2 Proposed Nested Stack Layout

```
activity-app/
  template.yaml                    # Root stack (orchestrator)
  functions/
    template.yaml                  # All 16 Lambda functions + ClientApi + ServerlessRestApi
  cdn/
    template.yaml                  # CloudFront distribution, cache policies, origin request policies,
                                   #   OAC, S3 bucket policy, CloudFront Function, DNS record
```

Two nested stacks, not three or four. Reasons:

- **Functions + API Gateways must stay together.** SAM `Events` blocks on Lambda functions generate API Gateway resources implicitly. The `ServerlessRestApi` (implicit) and `ClientApi` (explicit) are created by SAM based on the Event source annotations on functions. Splitting functions from their API Gateway is not feasible with SAM transforms.
- **CDN resources rarely change.** Cache policies, origin request policies, CloudFront distribution config, the CloudFront Function, OAC, bucket policy, and DNS record all change together when routing logic changes and almost never otherwise.
- **Two stacks is the sweet spot.** More granularity (e.g. separating DNS from CloudFront) adds cross-stack reference complexity for marginal benefit.

### 1.3 Root Stack: `activity-app/template.yaml`

The root stack becomes a thin orchestrator that:
1. Accepts all external parameters
2. Instantiates both nested stacks
3. Passes outputs from the functions stack into the CDN stack
4. Exports final outputs (CloudFront distribution ID, API endpoints, server domain)

**Parameters** (same as today):
- `Stage`
- `EnvironmentStackName`
- `BootstrapStackName`
- `ServerDomain`
- `HandleDomain`
- `ProxyDistributionId`
- `ActivityDistributionId`
- `ClientApiDomain`

**Resources:**

```yaml
Resources:
  FunctionsStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: functions/template.yaml   # sam package rewrites this to S3
      Parameters:
        Stage: !Ref Stage
        EnvironmentStackName: !Ref EnvironmentStackName
        BootstrapStackName: !Ref BootstrapStackName
        ServerDomain: !Ref ServerDomain
        HandleDomain: !Ref HandleDomain
        ProxyDistributionId: !Ref ProxyDistributionId
        ActivityDistributionId: !Ref ActivityDistributionId
        ClientApiDomain: !Ref ClientApiDomain
        CloudFrontDistributionId: !If
          - HasActivityDistribution
          - !Ref ActivityDistributionId
          - ""

  CdnStack:
    Type: AWS::CloudFormation::Stack
    # No explicit DependsOn needed — !GetAtt references create implicit dependency
    Properties:
      TemplateURL: cdn/template.yaml
      Parameters:
        Stage: !Ref Stage
        EnvironmentStackName: !Ref EnvironmentStackName
        BootstrapStackName: !Ref BootstrapStackName
        ServerlessRestApiId: !GetAtt FunctionsStack.Outputs.ServerlessRestApiId
        ClientApiId: !GetAtt FunctionsStack.Outputs.ClientApiId
        ProxyDistributionId: !Ref ProxyDistributionId
```

**Outputs** (from root, re-exporting nested outputs):

```yaml
Outputs:
  ApiEndpoint:
    Value: !GetAtt FunctionsStack.Outputs.ApiEndpoint
  CloudFrontDistributionId:
    Value: !GetAtt CdnStack.Outputs.CloudFrontDistributionId
    Export:
      Name: !Sub "${AWS::StackName}-CloudFrontDistributionId"
  CloudFrontDomainName:
    Value: !GetAtt CdnStack.Outputs.CloudFrontDomainName
  ServerDomainOutput:
    Value: !GetAtt FunctionsStack.Outputs.ServerDomainOutput
    Export:
      Name: !Sub "${AWS::StackName}-ServerDomain"
  ClientApiUrl:
    Value: !GetAtt FunctionsStack.Outputs.ClientApiUrl
    Export:
      Name: !Sub "${AWS::StackName}-ClientApiUrl"
```

### 1.4 Functions Stack: `activity-app/functions/template.yaml`

**Contains:**
- All 16 `AWS::Serverless::Function` resources
- `ClientApi` (`AWS::Serverless::Api`) — because PostFunction, MediaUploadFunction, and ProfileUpdateFunction reference it via `RestApiId: !Ref ClientApi`
- The implicit `ServerlessRestApi` — created by SAM from Event blocks that do not specify a RestApiId
- `Globals` section with shared Function defaults
- Its own `Conditions` block (replicating `IsProd`, `HasCrossDistribution`, etc. from the current flat template)
- Each nested template must be a complete, self-contained SAM template with its own `Transform: AWS::Serverless-2016-10-31`

**Implementation note on Globals:** The current `Globals` section uses `!ImportValue` with `Fn::Sub` for cross-stack references and `!If` conditions (`IsProd`). These conditions, imports, and the full `Globals` block must be replicated in the functions nested template along with its own `Conditions` block. Easy to miss during implementation.

**Parameters it receives from root:**
- `Stage`, `EnvironmentStackName`, `BootstrapStackName`, `ServerDomain`, `HandleDomain`
- `ProxyDistributionId`, `ActivityDistributionId`, `ClientApiDomain`
- `CloudFrontDistributionId` — so PostFunction and ProfileUpdateFunction can reference it for invalidation

**Key design note on CloudFront Distribution ID circular reference:**

Currently, PostFunction and ProfileUpdateFunction use `!Ref CloudFrontDistribution` for both:
1. The `CLOUDFRONT_DISTRIBUTION_ID` **env var**
2. The **IAM policy ARN** for `cloudfront:CreateInvalidation` (`!Sub "arn:aws:cloudfront::${AWS::AccountId}:distribution/${CloudFrontDistribution}"`)

In the nested architecture, the distribution lives in the CDN stack. Both the env var AND the IAM policy ARN must switch to the parameter-based distribution ID.

**Resolution:** The `ActivityDistributionId` parameter already exists and is passed as a workflow variable. The same pattern works here: PostFunction and ProfileUpdateFunction will use the `CloudFrontDistributionId` parameter passed from the root, which on first deploy uses `ActivityDistributionId` and on subsequent deploys uses the actual CDN output. The IAM policy ARNs become `!Sub "arn:aws:cloudfront::${AWS::AccountId}:distribution/${CloudFrontDistributionId}"`.

**First-deploy handling:** On first deploy, `CloudFrontDistributionId` is empty (no distribution exists yet). An empty string produces a malformed IAM ARN (`arn:aws:cloudfront::account:distribution/`) which fails CloudFormation validation. The functions template must use a `HasCloudFrontDistribution` condition (same pattern as the existing `HasActivityDistribution` / `HasCrossDistribution` conditions on InboxFunction) to conditionally include the invalidation IAM policy:

```yaml
Conditions:
  HasCloudFrontDistribution: !Not [!Equals [!Ref CloudFrontDistributionId, ""]]

# In PostFunction/ProfileUpdateFunction Policies:
- !If
  - HasCloudFrontDistribution
  - Statement:
      - Effect: Allow
        Action: cloudfront:CreateInvalidation
        Resource:
          - !Sub "arn:aws:cloudfront::${AWS::AccountId}:distribution/${CloudFrontDistributionId}"
  - !Ref AWS::NoValue
```

This means first deploy works (no invalidation permissions, no distribution to invalidate anyway), and the second deploy (after `ACTIVITY_DISTRIBUTION_ID` is set) adds the IAM policy. This is the exact same two-pass pattern already documented in Section 4.2 step 5-6.

**Implementation note on ServerDomainOutput:** The current template computes `ServerDomainOutput` as `!If [IsProd, !Ref ServerDomain, !Sub "${Stage}.${ServerDomain}"]`, while `Globals` computes `SERVER_DOMAIN` differently using `!ImportValue` from the bootstrap stack. These are two different computations that happen to produce the same result. In the nested architecture, unify this: compute the domain once in the root stack and pass it down as a parameter.

**Outputs it provides:**
- `ServerlessRestApiId` — `!Ref ServerlessRestApi`
- `ClientApiId` — `!Ref ClientApi`
- `ApiEndpoint` — `!Sub "https://${ServerlessRestApi}.execute-api.${AWS::Region}.amazonaws.com/Prod"`
- `ClientApiUrl` — `!Sub "https://${ClientApi}.execute-api.${AWS::Region}.amazonaws.com/${Stage}"`
- `ServerDomainOutput` — the computed server domain

### 1.5 CDN Stack: `activity-app/cdn/template.yaml`

**Contains:**
- `CloudFrontDistribution`
- All 5 cache policies (WebFinger, Outbox, Long, Medium, Short)
- All 3 origin request policies (Inbox, Session, ClientApi)
- `CloudFrontOAC`
- `MediaBucketPolicy`
- `ProfileRewriteFunction` (CloudFront Function)
- `FederationDnsRecord`

**Parameters it receives from root:**
- `Stage`, `EnvironmentStackName`, `BootstrapStackName`
- `ServerlessRestApiId` — used to construct API Gateway origin domain
- `ClientApiId` — used for ClientApiGateway origin
- `ProxyDistributionId`

**Outputs it provides:**
- `CloudFrontDistributionId` — `!Ref CloudFrontDistribution`
- `CloudFrontDomainName` — `!GetAtt CloudFrontDistribution.DomainName`

### 1.6 Cross-Stack Reference Summary

```
                         +-----------------+
                         |   Root Stack    |
                         | (orchestrator)  |
                         +--------+--------+
                                  |
              Parameters down     |     Outputs up
                                  |
           +----------------------+----------------------+
           |                                             |
  +--------v--------+                          +---------v-------+
  | Functions Stack  |   ServerlessRestApiId   |   CDN Stack     |
  |                  +------------------------->                 |
  | 16 Lambdas       |   ClientApiId           | CloudFront      |
  | ClientApi        +-------------------------> Cache Policies  |
  | ServerlessRestApi|                          | ORP, OAC        |
  |                  |                          | DNS, S3 Policy  |
  +------------------+                          +-----------------+
```

Data flows unidirectionally: Functions outputs feed into CDN inputs. No circular references between nested stacks.

### 1.7 CodeUri Paths

CodeUri paths in the functions template need to be relative to the functions template location. Since `swift package archive` outputs to `.build/plugins/AWSLambdaPackager/outputs/AWSLambdaPackager/{Handler}/{Handler}.zip`, the CodeUri from `activity-app/functions/template.yaml` would be `../../.build/plugins/AWSLambdaPackager/outputs/AWSLambdaPackager/{Handler}/{Handler}.zip`. With `sam package`, these get rewritten to S3 URIs.

## 2. Selective Builds for Stage

### 2.1 Current Build: Why It Is Slow

`swift package archive` compiles ALL executable products inside a Docker container. Even if only one handler changed, all are compiled and zipped. With the AWS SDK Swift dependency tree, a full build takes 15-20 minutes.

**Build caching is not sufficient.** Adding GitHub Actions `.build` cache reduced build times from ~21 minutes to ~18 minutes — a 3-minute improvement. SwiftPM incremental compilation helps marginally, but the archive plugin still links and zips all 18 products regardless. Selective builds are essential for `sam sync` to deliver meaningful speed improvements: there is no value in bypassing CloudFormation's 3-minute deploy if the build still takes 18 minutes.

### 2.2 Dependency Graph

From `Package.swift`:

**ActivityPubCore** (shared library) depends on: AWSDynamoDB, AWSSQS, AWSSSM, Crypto, _CryptoExtras

**Handlers that depend on ActivityPubCore:** all 16 except NodeInfoHandler and InstanceHandler.

**Additional dependencies beyond ActivityPubCore:**
- InboxHandler: +AWSCloudFront
- PostHandler: +AWSSSM, +AWSCloudFront
- MediaUploadHandler: +AWSS3, +AWSSSM
- ProfileUpdateHandler: +AWSS3, +AWSSSM, +AWSCloudFront
- ProfileHandler, AuthHandler, ComposeHandler: +Elementary

### 2.3 Selective Build Strategy

**Step 1: Determine changed paths via git diff.**

```bash
CHANGED_FILES=$(git diff --name-only $LAST_DEPLOYED_SHA HEAD)
```

**Step 2: Map changed files to Swift targets.**

- `Sources/ActivityPubCore/**` → ActivityPubCore changed → rebuild all 16 dependents
- `Sources/{HandlerName}/**` → rebuild that handler only
- `Package.swift` or `Package.resolved` → rebuild all (dependency change)
- `docker/Dockerfile.al2023-swift` → rebuild all (base image change)

**Step 3: Build only changed targets inside Docker.**

`swift package archive` does not support building individual products. Instead, build manually inside the Docker container:

```bash
docker run --rm \
  -v "$(pwd):/workspace" \
  -v "$(pwd)/.build:/workspace/.build" \
  -w /workspace \
  swift-al2023:6.3 \
  bash scripts/build-selective.sh $TARGETS_TO_BUILD
```

Where `build-selective.sh` runs `swift build -c release --product $TARGET --static-swift-stdlib` for each target and zips the output.

**Step 4: For unchanged targets, reuse existing zips from cache.**

Cache the zip output directory. On selective builds, only overwrite zips for targets that were rebuilt. `sam package` uploads all zips to S3, but only changed ones have new hashes, so CloudFormation only updates Lambda functions whose CodeUri changed.

**Step 5: Fall back to full build when needed.**

If `Package.swift`, `Package.resolved`, or `docker/Dockerfile.al2023-swift` changed, fall back to `swift package archive` (full build).

### 2.4 Tracking Deployed Commits

After a successful deploy, tag the deployed commit:

```bash
git tag -f "deploy-${STAGE}" HEAD
git push -f origin "deploy-${STAGE}"
```

## 3. sam sync for Stage Deploys

### 3.1 What sam sync Provides

`sam sync --code` bypasses CloudFormation entirely for Lambda code updates. It directly calls `lambda:UpdateFunctionCode` for each function, completing in seconds instead of minutes.

### 3.2 Deploy Strategy by Stage

**Stage deploys:**
- Code-only change (only `Sources/` modified) → `sam sync --code`
- Infrastructure change (template YAML, Package.swift, etc.) → full `sam deploy`

**Prod deploys:** Always `sam deploy`. Never `sam sync`.

### 3.3 Workflow Changes

New workflow: `.github/workflows/fast-stage-deploy.yml`

```yaml
name: Fast Stage Deploy
on:
  push:
    branches: [main]
    paths:
      - 'Sources/**'
```

This builds only changed targets and uses `sam sync --code` to push them. Completes in minutes instead of 20+.

The existing `build.yml` + `deploy.yml` pipeline continues for full deploys (template changes, prod releases).

### 3.4 sam sync Command

```bash
sam sync --code \
  --stack-name "activity-app-stage" \
  --template-file activity-app/template.yaml \
  --parameter-overrides Stage=stage ... \
  --region us-east-1 \
  --no-watch
```

**Important:** `--no-watch` prevents entering watch mode in CI.

### 3.5 Direct Lambda Update (Primary Fast Path)

`sam sync --code` is documented to work with nested stacks but is unreliable in practice — it can fail to resolve Lambda physical resource IDs across nested stacks, particularly with SAM transforms. **The primary fast path should be direct `aws lambda update-function-code` calls**, not `sam sync`.

The fast-stage-deploy workflow calls `aws lambda update-function-code` directly for each changed function. Function names are resolved from the CloudFormation stack (not derived via string manipulation) to avoid fragile naming assumptions:

```bash
# Build a target-to-function-name map from the stack's Lambda resources
FUNCTION_MAP=$(aws cloudformation list-stack-resources \
  --stack-name "activity-app-stage" \
  --query "StackResourceSummaries[?ResourceType=='AWS::Lambda::Function'].[LogicalResourceId,PhysicalResourceId]" \
  --output text)

for TARGET in $CHANGED_TARGETS; do
  # Map Swift target name (e.g. "PostHandler") to logical ID (e.g. "PostFunction")
  LOGICAL_ID="${TARGET/Handler/Function}"
  FUNCTION_NAME=$(echo "$FUNCTION_MAP" | grep "$LOGICAL_ID" | awk '{print $2}')

  aws lambda update-function-code \
    --function-name "$FUNCTION_NAME" \
    --zip-file "fileb://.build/lambda/$TARGET/$TARGET.zip" \
    --region us-east-1
done
```

Note: with nested stacks, the Lambda resources are in the functions child stack, so the query targets the nested stack's physical stack name. The root stack's resources are `AWS::CloudFormation::Stack` types, not Lambda functions.

This bypasses both CloudFormation and SAM entirely. Combined with selective builds, a single-handler change goes from 20+ minutes to under 2 minutes: build one target (~1 min) + upload one zip (~5 sec).

`sam sync --code` can be attempted as an alternative, but direct Lambda update is the reliable path.

## 4. Migration Path

### 4.1 The Core Problem

CloudFormation does not support "move resource X from the root stack into a nested stack." A flat-to-nested change means CloudFormation sees resources being deleted and nested stack resources being created. This would recreate Lambda functions (new ARNs), CloudFront distribution (new ID, 15-min propagation), DNS records, etc.

### 4.2 Stage Migration: Destructive (Safe)

Stage can tolerate downtime. Delete and redeploy.

1. Record current state: `aws cloudformation describe-stacks --stack-name activity-app-stage`
2. Delete: `aws cloudformation delete-stack --stack-name activity-app-stage`
3. Wait: `aws cloudformation wait stack-delete-complete --stack-name activity-app-stage`
4. Deploy new nested stack:
   ```bash
   sam deploy --template-file activity-app/template.yaml \
     --stack-name activity-app-stage \
     --capabilities CAPABILITY_IAM CAPABILITY_AUTO_EXPAND \
     --parameter-overrides Stage=stage ...
   ```
5. Get new CloudFront distribution ID, update `ACTIVITY_DISTRIBUTION_ID` variable
6. Redeploy to pick up distribution ID
7. Reprovision stage actor
8. Run integration tests

**Rollback:** Redeploy flat template from git history.

### 4.3 Prod Migration: Blue/Green DNS Cutover

1. **Deploy parallel stack** as `activity-app-prod-v2` sharing `activity-environment-prod`
2. **Handle alias conflict:** Two CloudFront distributions cannot share the same CNAME. New stack uses temporary alias (e.g., `v2.activity.happitec.com`) during testing. The CDN template needs a `DomainOverride` parameter for this.
3. **Test** all endpoints against the new stack using its CloudFront domain
4. **DNS cutover — scripted as a tight atomic operation:**
   The window between removing the alias from old CF and adding to new CF is the riskiest moment. CloudFront rejects requests for domains not in its alias list, so this is not a DNS TTL issue — it's a CloudFront-level gap. Script this as `scripts/prod-alias-swap.sh`:
   ```bash
   #!/bin/bash
   set -euo pipefail
   OLD_CF_ID=$1
   NEW_CF_ID=$2
   ALIAS="activity.happitec.com"

   # Pre-stage: get both distribution configs
   OLD_CONFIG=$(aws cloudfront get-distribution-config --id "$OLD_CF_ID")
   NEW_CONFIG=$(aws cloudfront get-distribution-config --id "$NEW_CF_ID")

   # Step 1: Remove alias from old distribution
   # (modify OLD_CONFIG to remove ALIAS from Aliases.Items, update)
   echo "Removing alias from old distribution..."
   aws cloudfront update-distribution --id "$OLD_CF_ID" ...

   # Step 2: Immediately add alias to new distribution
   echo "Adding alias to new distribution..."
   aws cloudfront update-distribution --id "$NEW_CF_ID" ...

   # If step 2 fails, rollback step 1
   # (re-add alias to old distribution)
   ```
   The script must handle failure of step 2 by automatically rolling back step 1 (re-adding the alias to the old distribution). Run during a low-traffic window.
5. **Update GitHub variables:** `ACTIVITY_DISTRIBUTION_ID`, `CLIENT_API_DOMAIN_PROD`
6. **Delete old stack:** `aws cloudformation delete-stack --stack-name activity-app-prod`
7. **Reclaim stack name (optional):** Deploy new template as `activity-app-prod`, then delete `activity-app-prod-v2`. Or keep `-v2` and update workflow variables — the CFN name is just a string.

**Federation URL preservation:** All URLs use the domain name, not CloudFront domain. As long as DNS resolves to a distribution with the correct Lambda functions and DynamoDB access, everything works.

**Rollback:** Old stack is running until explicitly deleted. Run the alias swap script in reverse to point back.

### 4.4 Named Resource Collisions

All named CloudFront resources must be unique per AWS account. During prod migration, both stacks would try to create resources with the same names. **Fix:** use `${AWS::StackName}` instead of `${Stage}` in ALL named resources:

| Current Name Pattern | Resource Type |
|---------------------|---------------|
| `activity-webfinger-cache-${Stage}` | Cache Policy |
| `activity-outbox-cache-${Stage}` | Cache Policy |
| `activity-long-cache-${Stage}` | Cache Policy |
| `activity-medium-cache-${Stage}` | Cache Policy |
| `activity-short-cache-${Stage}` | Cache Policy |
| `activity-inbox-origin-${Stage}` | Origin Request Policy |
| `activity-session-origin-${Stage}` | Origin Request Policy |
| `activity-clientapi-origin-${Stage}` | Origin Request Policy |
| `activity-media-oac-${Stage}` | Origin Access Control |
| `activity-profile-rewrite-${Stage}` | CloudFront Function |

All must switch to `${AWS::StackName}` (or a derived suffix) for the parallel-stack migration to work. This change should be made in Phase 1 when creating the CDN nested template.

## 5. Ephemeral PR Environments

### 5.1 Design

Each PR deploys the **functions stack only** (Lambda functions + API Gateways), sharing the stage environment stack. No CloudFront, no DNS, no cache policies.

- Stack name: `activity-app-pr-{PR_NUMBER}`
- Access via raw API Gateway execute-api URL
- `SERVER_DOMAIN` set to `stage.activity.happitec.com` (functional correctness, not domain routing)

### 5.2 Deployment

```yaml
# .github/workflows/pr-deploy.yml
on:
  pull_request:
    types: [opened, synchronize]
```

Builds changed targets, deploys `activity-app/functions/template.yaml` as a standalone stack.

### 5.3 Lambda Naming

Current: `activity-app-post-${Stage}`. PR envs would collide with stage. **Fix:** add `StackSuffix` parameter defaulting to empty, set to `-pr-{N}` for PR deploys. Function names become `activity-app-post-${Stage}${StackSuffix}`.

### 5.4 Teardown

```yaml
# .github/workflows/pr-cleanup.yml
on:
  pull_request:
    types: [closed]
```

Deletes `activity-app-pr-{N}`. Add scheduled cleanup for stale stacks.

### 5.5 Isolation

PR environments share stage DynamoDB. Risks:
- Data written by PR Lambdas is visible to stage
- A buggy PR Lambda could write malformed data that breaks stage

Mitigations:
- Provision PR actors with a `pr-{N}-` prefix (e.g., `pr-42-testuser`) so PR data is visually distinguishable and easily cleaned up
- PR Lambdas use the stage `SERVER_DOMAIN`, so generated ActivityPub URLs resolve correctly for functional testing
- Stronger isolation (DynamoDB key prefixing or separate tables) deferred to a later phase

## 7. Risks and Open Questions

### Risks

1. **SAM Transform in nested stacks.** Each nested template needs `Transform: AWS::Serverless-2016-10-31`. Verify `sam sync` resolves nested SAM transforms.

2. **`sam sync --code` with nested stacks.** Documentation is ambiguous. Fallback: call `aws lambda update-function-code` directly.

3. **CloudFront alias conflict during prod migration.** Brief window where neither distribution has the alias. Keep window short, do during low traffic.

4. **Cache policy name collisions.** Use `${AWS::StackName}` instead of `${Stage}` in policy names.

5. **Selective build correctness.** If the dependency graph in `detect-changed-targets.sh` diverges from `Package.swift`, a handler might not be rebuilt when it should. The failure mode is silent and dangerous (deploying stale code). Mitigations: (a) parse `Package.swift` programmatically rather than hardcoding the graph, (b) add a CI validation step that checks the script's graph matches `Package.swift`, (c) always fall back to full build on any ambiguity.

### Open Questions

1. **Stack naming for prod:** Keep `activity-app-prod-v2` permanently or cycle through to reclaim `activity-app-prod`? Recommendation: reclaim the name for consistency across variables, scripts, and mental models.

2. **Fast deploy workflow architecture:** The fast-stage-deploy path needs both the source code (for selective build) and AWS credentials (for `aws lambda update-function-code`). This may need to be a single job rather than the current build/deploy split, since `sam sync` and direct Lambda updates need the built zips locally.

3. **PR environment Lambda naming:** `StackSuffix` parameter prevents collisions but adds complexity to every function name. Worth the trade-off for PR envs?

4. **Cross-invalidation in PR environments:** Lambda code must handle empty `CLOUDFRONT_DISTRIBUTION_ID` gracefully (skip invalidation). Verify this is already the case.

5. **DeliverFunction SQS event source:** DeliverFunction uses an SQS event type (not API Gateway), referencing `${EnvironmentStackName}-QueueArn` via ImportValue. This works in nested stacks but is a special case worth noting — it's the only function not triggered by API Gateway.

### Resolved Questions

- **Selective build vs. build cache:** RESOLVED. `.build` cache reduced builds from 21 to 18 minutes — insufficient. Selective builds are required for the fast path to deliver meaningful improvement.
